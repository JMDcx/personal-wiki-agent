## 当前任务背景

项目是一个飞书里的知识沉淀与知识问答 Agent。用户主要通过飞书和机器人聊天完成两类事情：

- 把排查过程、链接、图片、文档等内容沉淀到飞书知识库。
- 后续通过机器人从已沉淀知识里问答。

第一个优化是：建立 Agent/RAG 离线评测与回归体系。目标不是写一个简单脚本，而是建立后续 prompt、RAG、Qdrant、rerank、controller、模型配置修改前后的质量验证闭环。

## 已实现能力

### 1. Agent eval

命令：

```powershell
.venv\Scripts\python.exe scripts\replay_eval_dataset.py `
  --mode agent `
  --dataset tests\fixtures\evals\smoke_cases.jsonl
```

能力：

- 调用真实 `invoke_agent`。
- 每条 case 用独立 thread。
- 读取结构化日志中的 intent、allow_retrieval、rewrite、timing。
- 输出 `summary.json`、`results.jsonl`、`report.md`。

### 2. Retrieval-only eval

命令：

```powershell
.venv\Scripts\python.exe scripts\replay_eval_dataset.py `
  --mode retrieval `
  --dataset data\evals\local_retrieval_smoke.jsonl
```

能力：

- 直接调用 `RAGQueryPipeline.prepare_context`。
- 不走最终 LLM 生成。
- 检查 source hit、match_status、context must include、retrieval latency。
- 后续用于混合检索、rerank、引用校验优化前后的质量对比。

### 3. Baseline 对比

命令：

```powershell
.venv\Scripts\python.exe scripts\replay_eval_dataset.py `
  --mode retrieval `
  --dataset data\evals\local_retrieval_smoke.jsonl `
  --baseline data\evals\runs\<old_run_id>\summary.json
```

能对比：

- pass rate
- source hit rate
- match_status accuracy
- answer match rate
- p95 latency
- newly failed cases
- fixed cases
- still failed cases

## 关于 chunk 的讨论结论

当前 chunking 配置：

```text
chunk_size = 512
chunk_overlap = 128
```

这是字符级切分，不是 token 或中文分词切分。

当前 26 个 chunk 的长度分布曾统计为：

```text
min = 286
avg = 741.2
median ≈ 473
max = 2955
```

建议目标：

```text
普通知识文档：400-800 中文字符 / chunk
排查步骤类文档：500-1000 中文字符 / chunk
配置/API/代码块文档：允许偶尔 1000-1500 字符
超过 2000 字符：后续需要重点检查
```

当前不要急着改 chunk size。等知识库扩到 100+ chunk 后，再用 retrieval-only eval 对比 `512/128` 和 `800/160`。

## 关于知识库规模的讨论结论

当前 1 个文档、26 个 chunk 太小。

它适合：

- 验证 eval 框架
- 验证 Qdrant 是否能查
- 验证报告、baseline、badcase 导出是否正常

它不适合：

- 做 20-50 条正式 regression case
- 严肃评估 rerank
- 严肃评估混合检索
- 证明检索召回质量

推荐知识库规模：

```text
最低可用：1 万字左右
推荐起步：3 万字以上
比较理想：5 万到 8 万字
10 万字以上：适合更系统的检索优化
```

推荐先补：

- 项目使用手册
- 知识沉淀流程
- 知识问答流程
- 飞书机器人配置、权限、事件订阅
- Qdrant 启动、索引、重建、排查
- 常见 badcase 和修复记录
- 真实 debug 记录
- 已完成优化记录
- API/环境变量说明

## 关于 regression case 的讨论结论

不要再用“这个知识库支持什么能力？”这种产品说明页式问题作为核心 case。它不太像用户在飞书里和机器人聊天。

更贴近产品定位的问题应该是：

- 我之前沉淀过的登录失败问题，最后是怎么解决的？
- 上次那个回调地址配置错误的问题，排查步骤是什么？
- 有没有记录过飞书机器人收不到消息的排查方法？
- 知识库里有没有提到如何重新构建索引？
- 帮我把这段排查过程沉淀到知识库：用户登录失败，最后发现是回调地址配置错误。
- 把这个问题记录一下：Qdrant 容器没启动会导致检索失败，解决方式是先启动 qdrant 服务。
- 那这个问题后来怎么修的？
- 知识库里有没有记录火星基地报销制度？
- 你好
- 你是谁？

正式第一批建议：

```text
3 条知识问答
3 条知识沉淀
2 条追问
1 条无答案拒答
1 条不该检索
```

## 需要用户人工介入的地方

正式 regression dataset 必须基于真实知识库内容，不能靠模型猜。

需要人工确认：

- 哪些问题是真实飞书聊天场景。
- 每个问题知识库里是否真的有答案。
- 正确来源标题或 source URI 是什么。
- 答案必须包含哪些信息点。
- 答案不能包含哪些错误断言。
- 无答案时的拒答话术。
- 知识沉淀请求的 intent 名称和是否禁止检索。
